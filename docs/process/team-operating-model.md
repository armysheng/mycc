# MyCC 团队运行手册（V2）

> 目标：在多人并行开发下，降低冲突、降低返工、提高交付可预测性。

## 1. 适用范围

- 适用于 `mycc` 仓库所有功能开发、修复、回归、发布协作。
- 角色包含：`human`、`codex`、`claude`、`andy`、`mycc_test`。

## 2. 角色与职责

- `human`：产品范围、优先级、取舍、最终放行。
- `codex`：任务拆解、接口/架构裁决、门禁检查、冲突裁决。
- `claude`：按任务实现、提交 PR、修复 review 问题。
- `andy`：按任务实现、提交 PR、修复 review 问题。
- `mycc_test`：回归执行、缺陷报告、验收结论。

## 3. 协作基础规则

- 一人一个 worktree，禁止多人共用同一目录开发。
- 一任务一个 owner，一任务一个分支，一任务一个 PR。
- `main` 只用于集成与验收，不在主干目录直接开发功能。
- 任何“完成”结论必须落地到文档或看板，口头不算完成。

## 4. 分支、目录、端口规范

- 分支前缀：`codex/*`、`cc/*`。
- 若需区分个人，使用 `codex/<name>-<topic>`，例如 `codex/andy-skill-fix`。
- 目录建议：
  - `main`: `/Users/armysheng/workspace/mycc`
  - `andy`: `/Users/armysheng/workspace/mycc_andy`
  - `claude`: `~/.claude/worktrees/...`
- 端口约定：`main=3000/8080`，`codex=3001/8081`，`claude=3002/8082`。

## 5. 开发生命周期（统一流程）

1. 立项：在看板登记任务，明确 owner、范围、验收标准。
2. 设计：输出最小设计说明（接口、数据流、风险点）。
3. 开发：在个人 worktree 实现，不跨任务混改。
4. 自测：至少完成 build + 关键冒烟。
5. 提 PR：填写变更范围、风险、回归步骤、回滚方式。
6. 评审：按严重级别处理问题，P0/P1 必须清零。
7. 合并：满足门禁后合并。
8. 回归：测试执行对应分层用例并回写结果。

## 6. 质量门禁（Merge Gate）

合并前必须全部满足：

1. CI 全绿。
2. PR 描述完整（范围、风险、回归、回滚）。
3. build 通过（前后端受影响模块）。
4. 核心冒烟通过。
5. 无未决 P0/P1。
6. 已明确测试回归范围与负责人。

## 7. 缺陷严重级别（Review 统一口径）

- P0：阻塞发布/核心链路不可用/数据错误，必须立即修。
- P1：高风险功能错误，合并前必须修。
- P2：中风险问题，可当期修或跟进修。
- P3：优化建议，不阻塞合并。

## 8. 测试分层与执行策略

- PR 阶段：最小冒烟（登录、onboarding、聊天、技能页主链路）。
- main 阶段：冒烟 + 核心回归。
- 夜间阶段：全量回归（允许 flaky 重试一次，保留 trace/screenshot）。
- 阻塞用例需写明“阻塞原因 + 缺失前置条件”，不能算失败。

## 9. 环境与配置管理

- 新建 worktree 必做：复制 `.env`、安装依赖、验证端口、跑一次 build。
- `.env` 不入库，环境差异必须在 PR 描述里明确。
- 模板/目录依赖（如 `/opt/mycc/templates/user-workspace`）变更需同步记录。

## 10. 交接机制（每日强制）

每位执行者收工前提交“交接四件套”：

1. 分支与 commit SHA。
2. 已完成 / 未完成。
3. 风险与阻塞。
4. 下一步。

## 11. 紧急修复（Hotfix）

- 可先修后补流程，但 24 小时内必须补齐：看板记录、PR 说明、回归证据。
- 若 hotfix 涉及策略变更，必须补更新本手册与相关模板。

## 12. 规范变更机制

- 规范由 `codex` 维护版本，`human` 最终确认。
- 每次新增成员或流程升级时，至少同步更新：
  - `team-collaboration-contract.md`
  - `team-operating-model.md`
  - `new-member-onboarding.md`
